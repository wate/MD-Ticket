AIエージェント向けガイド
=========================

このドキュメントは、AIエージェントがこのチケット管理システムを効果的に利用するためのガイドラインです。

重要な前提条件
-------------------------

### このドキュメントを参照するための設定

このドキュメント（`.ticket/AGENTS.md`）は独立して存在しますが、AIエージェントが自動的に参照する仕組みはありません。このチケット管理システムを正しく機能させるためには、プロジェクトルートの`AGENTS.md`にチケット管理情報を統合する必要があります。

統合方法の詳細は[README.md](README.md)の「AGENTS.md統合ガイド」セクションを参照してください。

### 統合しない場合のリスク

- AIがMD-Ticketの存在を認識できない
- チケット作成時に誤った場所（`docs/`など）にファイルを作成してしまう
- テンプレート構造や命名規則が守られない
- GitHub Copilot以外のAI（Claude、ChatGPTなど）では使用できない

目的とスコープ
-------------------------

このガイドは、AIエージェントが以下の作業を適切に実行するための指針を提供します。

- ユーザーの要求やアイデアを適切なチケット種別に分類する
- テンプレートを使用して一貫性のあるチケットを作成する
- 既存チケットの更新や管理を支援する
- ユーザーに確認が必要な事項を判断する

基本原則
-------------------------

1. **明確性の優先**: 曖昧な情報や不足している情報がある場合は、チケットを作成する前にユーザーに確認する
2. **テンプレートの遵守**: 各チケット種別のテンプレート構造を必ず使用する
3. **最小限の推測**: ユーザーが明示していない情報を推測で補わない
4. **トレーサビリティ**: チケット間の関連性を明確に記録する
5. **段階的な詳細化**: アイデア→要望→タスクという流れを尊重する
6. **ライフサイクル管理**: チケットの状態変化に応じてファイルを移動・削除する

バージョン管理について
-------------------------

このチケット管理システムは、バージョン管理システム(Git等)の使用を前提としていません。  
履歴管理の必要性は、利用するユーザーが判断します。

### Git管理する場合

- ファイルの移動・削除の履歴が追跡できる
- 削除したチケットを後から参照・復元できる
- チームでの共有や変更履歴の確認が容易

### Git管理しない場合

- よりカジュアルに、メモ感覚で使用できる
- 削除したファイルは復元できない
- 個人用途やシンプルな運用に適している

エージェントは、Git管理の有無を意識する必要はありません。  
ユーザーが管理方法を選択します。

設定ファイルの参照
-------------------------

チケット種別の情報は `.ticket/config.yml` で管理されている場合があります。

### 設定ファイルがある場合

`.ticket/config.yml`が存在する場合、以下の情報を参照してチケット作成を行います。

```yaml
types:
  bug:
    label: バグ報告
    template: bug.md
    description: 不具合の報告
  task:
    label: タスク
    template: task.md
    description: 実行すべき作業
  # ... 他の種別やカスタム種別
```

設定ファイルから以下を確認します。

- 利用可能なチケット種別（キー名がディレクトリ名に対応）
- 各種別の表示名（`label`）
- 使用するテンプレート（`template`）
- 種別の説明（`description`）

### 設定ファイルがない場合

デフォルトの4種別を使用します。

- `bug`: バグ報告
- `task`: タスク
- `idea`: アイデア
- `request`: 要望

### チケット作成時の判断フロー

1. `.ticket/config.yml`の存在を確認
2. 存在する場合、設定ファイルから種別情報を取得
3. 存在しない場合、デフォルト4種別を使用
4. ユーザーの依頼内容から適切な種別を判断
5. 対応するテンプレートを使用してチケットを作成

### カスタム種別の追加

プロジェクト固有のチケット種別を追加する場合の手順です。

#### 1. 設定ファイルに種別定義を追加

`.ticket/config.yml`に種別を追加します。

```yaml
types:
  bug:
    label: バグ報告
    template: bug.md
    description: 不具合の報告
  task:
    label: タスク
    template: task.md
    description: 実行すべき作業
  idea:
    label: アイデア
    template: idea.md
    description: 発想・構想段階のメモ
  request:
    label: 要望
    template: request.md
    description: 実現してほしい希望
  spike:
    label: スパイク
    template: spike.md
    description: 技術調査タスク
```

#### 2. ディレクトリを作成

```bash
mkdir .ticket/spike
```

#### 3. テンプレートファイルを作成

```bash
# 既存テンプレートをベースに作成
cp .ticket/_template/task.md .ticket/_template/spike.md
# 内容を編集
```

チケット種別の判断基準
-------------------------

### アイデア(`idea/`)

以下の条件に該当する場合、アイデアとして記録します。

- 具体的な実装方法が定まっていない発想や構想
- 「こんなことができたら」「もしかしたら」といった仮説段階の内容
- 技術的な実現可能性が未検証の提案
- まだ誰にも相談していない個人的なひらめき

#### 判断に迷うケース

- ユーザーが「考えているんだけど」「思いついた」などの表現を使った場合 -> アイデア
- 実現したい目的は明確だが、手段が不明確な場合 -> アイデア

### 要望(`request/`)

以下の条件に該当する場合、要望として記録します。

- 「こうなってほしい」という希望や期待が明確
- 具体的な作業には至らないが、方向性や目標が定まっている
- 他者への依頼や提案を前提とした内容
- 実現方法は後で検討するが、やりたいことは明確

#### 判断に迷うケース

- 「できたらいいな」と「やってほしい」の中間の場合 -> ユーザーに「すぐに着手する予定か」を確認
- 具体的な期限がない場合 -> 要望(期限付きの場合はタスク候補)

### タスク(`task/`)

以下の条件に該当する場合、タスクとして記録します。

- 実行すべき具体的な作業内容が明確
- 前提条件や必要なリソースが特定できている
- 完了条件や成果物が定義可能
- 誰かが実際に着手する段階にある

#### 判断に迷うケース

- アイデアや要望から派生した実行可能な作業 -> 元のチケットへの参照を含めてタスク化
- 複数の小さなタスクに分解できそうな大きな作業 -> ユーザーに分割の必要性を確認

### バグ(`bug/`)

以下の条件に該当する場合、バグとして記録します。

- 既存のシステムやプロセスの不具合
- 期待される動作と実際の動作に明確な乖離がある
- 再現可能な問題がある
- 修正が必要な誤動作や予期しない挙動

#### 判断に迷うケース

- 仕様が不明確で、バグなのか仕様なのか判断できない場合 -> ユーザーに仕様を確認
- パフォーマンス問題や使いにくさの報告 -> 重大な場合はバグ、改善要望の場合は要望

チケット作成のワークフロー
-------------------------

### 1. 情報の確認

チケット作成前に、以下の情報が揃っているか確認します。

#### 共通項目

- チケットの種別を判断できる十分な情報
- 背景や理由(なぜこのチケットが必要か)

#### 種別固有の必須項目

- アイデア: きっかけ、概要・ひらめきの内容
- 要望: 概要、背景・理由
- タスク: 前提条件、依頼/作業内容、背景・理由
- バグ: 不具合の概要、再現手順、期待される動作、実際の動作

不足している情報がある場合は、具体的に何が必要かをユーザーに伝えます。

### 2. ファイル名の決定

ファイル名は以下の形式で作成します。

```
brief-description.md
```

- `brief-description`: 内容を簡潔に表す英数字とハイフン(例: implement-user-auth)
- 日本語の内容の場合もファイル名は英語で記述
- 単語はハイフンで区切る(アンダースコア不可)
- 全て小文字を使用

#### 例

- `add-search-feature.md`
- `fix-login-bug.md`
- `improve-performance.md`

### 3. テンプレートの使用

1. `_template/` ディレクトリから適切なテンプレートを取得
2. テンプレート内の各項目を埋める
3. オプション項目は、情報がない場合は削除可(空のまま残さない)
4. 必須項目は必ず記入(情報がない場合はユーザーに確認)

### 4. チケットの保存

適切なディレクトリに保存します。

- アイデア: `idea/brief-description.md`
- 要望: `request/brief-description.md`
- タスク: `task/brief-description.md`
- バグ: `bug/brief-description.md`

チケットの更新と管理
-------------------------

### 既存チケットの更新

以下の場合、既存チケットの更新を提案します。

- 追加情報が得られた場合
- 進捗状況の記録が必要な場合
- 関連する新しいチケットが作成された場合

更新時は以下の点に注意します。

- チケットの内容を直接更新する
- 大幅な変更の場合は、ユーザーに確認を求める

### チケットの種別変更(ファイル移動)

チケットの成熟度に応じて、種別を変更する必要がある場合があります。

- アイデア -> 要望: 具体的にやりたいことが明確になった
- アイデア/要望 -> タスク: 実行可能な作業に落とし込めた
- タスク -> バグ: 実装後に不具合として扱うべきことが判明した

#### ファイル移動の手順

1. ユーザーに種別変更の確認を取る
2. ファイルを適切なディレクトリに移動

   ```bash
   # 例: 要望からタスクへ
   mv request/add-feature.md task/add-feature.md
   ```

3. 必要に応じてテンプレート項目を更新(種別に応じた必須項目の追加など)
4. 他のチケットから参照されている場合は、パスを更新

#### 注意事項

- ファイル名は変更しない(説明部分を維持)
- ディレクトリのみを変更する

### チケットの完了

タスクやバグが完了した場合、以下の手順で処理します。

1. ユーザーに完了の確認を取る
2. 完了の記録が必要な場合は、ADRや成果物への参照を確認
3. ユーザーにチケットの処理方法を確認する

   ```
   チケット「task/completed-task.md」が完了しました。
   どのように処理しますか？
   
   1. 削除する（推奨）
   2. アーカイブする（後で参照したい場合）
   ```

4. 選択に応じて処理を実行する
    - 削除の場合: ファイルを削除
    - アーカイブの場合: [アーカイブの手順](#チケットのアーカイブ)に従う

#### 削除前に確認すべきこと

- 成果物や結果が適切に記録されているか
- 他のチケットから参照されていないか
- 将来参照する必要がある情報は別途記録されているか

### チケットの取りやめ

アイデアや要望、タスクを取りやめる場合。

1. 取りやめの理由をユーザーに確認
2. 重要な判断の場合、ADRとして記録することを提案
3. ユーザーにチケットの処理方法を確認する

   ```
   チケット「idea/cancelled-idea.md」を取りやめます。
   どのように処理しますか？
   
   1. 削除する（推奨）
   2. アーカイブする（判断経緯を残したい場合）
   ```

4. 選択に応じて処理を実行する
    - 削除の場合: ファイルを削除
    - アーカイブの場合: [アーカイブの手順](#チケットのアーカイブ)に従う

#### 取りやめの記録

取りやめの理由が今後の参考になる場合は、ADRとして記録することを推奨します。

```
このアイデアを取りやめる判断は、今後の参考になる可能性があります。
ADRとして記録しますか？

### 理由

- 技術的制約が明確になった
- コストとベネフィットのバランスが合わない
- 方針転換により不要になった
```

### チケットのアーカイブ

完了したチケットのうち、後で参照したいものをアーカイブします。

#### アーカイブすべきチケット

- 完了したチケットの中で、判断経緯や検討内容を残しておきたいもの
- 破棄したアイデアの中で、将来的に参考になる可能性があるもの
- 後から「なぜこうなったか」を振り返る可能性があるもの

#### アーカイブの手順

1. アーカイブ先ディレクトリを決定

   ```bash
   # 現在の年月を取得（例: 2025-01）
   ARCHIVE_DIR="_archive/$(date +%Y-%m)"
   mkdir -p "$ARCHIVE_DIR"
   ```

2. チケットファイルを移動（種別プレフィックスを付与）

   ```bash
   # 例: task/completed-task.md → _archive/2025-01/task-completed-task.md
   mv task/completed-task.md "$ARCHIVE_DIR/task-completed-task.md"
   
   # 例: idea/cancelled-idea.md → _archive/2025-01/idea-cancelled-idea.md
   mv idea/cancelled-idea.md "$ARCHIVE_DIR/idea-cancelled-idea.md"
   ```

3. 参考資料がある場合は移動

   ```bash
   # チケットに関連する参考資料をアーカイブ
   mv _files/completed-task-*.* _archive/_files/
   ```

4. 他のチケットから参照されている場合は、参照元を更新

   ```md
   <!-- 更新前 -->
   関連: [completed-task.md](../task/completed-task.md)
   
   <!-- 更新後（アーカイブ済みを明示） -->
   関連: [completed-task.md](../_archive/2025-01/task-completed-task.md)（アーカイブ済み）
   ```

5. ファイル名が他の種別のチケットと衝突する場合

   アーカイブ時に種別プレフィックスを付与するため、異なる種別で同じ名前のチケットがアーカイブされても衝突しません。

   ```bash
   # これらは同じアーカイブディレクトリに共存可能
   _archive/2025-01/idea-new-feature.md
   _archive/2025-01/task-new-feature.md
   _archive/2025-01/request-new-feature.md
   ```

6. 参考資料が衝突する場合

   参考資料は元々チケット名をプレフィックスとして持つため、通常は衝突しません。
   万が一衝突する場合は、ユーザーに報告して指示を仰ぎます。

   ```
   参考資料「_files/new-feature-draft.md」をアーカイブしようとしましたが、
   「_archive/_files/new-feature-draft.md」が既に存在します。
   
   どのように処理しますか？
   1. 上書きする
   2. 別名で保存する（例: new-feature-draft-2.md）
   3. 新しいファイルを削除する
   ```

#### アーカイブ後のリンクパス

参考資料も一緒に `_archive/_files/` に移動するため、チケット内の相対パスは変更不要です。

```md
<!-- アーカイブ前: task/completed-task.md -->
参考資料
-------------------------

- [ドラフト](../_files/completed-task-draft.md)

<!-- アーカイブ後: _archive/2025-01/task-completed-task.md -->
<!-- 参考資料も _archive/_files/ に移動しているため、パスはそのまま -->
参考資料
-------------------------

- [ドラフト](../_files/completed-task-draft.md)
```

チケットと参考資料の階層構造が保たれるため、リンクは正しく動作します。

### チケット間の関連付け

関連するチケットがある場合、以下の形式で記述します。

```md
関連チケット
-------------------------

- 関連: [related-ticket.md](../category/related-ticket.md) - 簡単な説明
- 前提: [prerequisite.md](../task/prerequisite.md) - このタスクの前提となる作業
- 派生: [derived-task.md](../task/derived-task.md) - このチケットから派生したタスク
```

#### 関連タイプ

- **関連**: 内容的に関係があるチケット
- **前提**: このチケットの前提条件となるチケット
- **派生**: このチケットから生まれた新しいチケット
- **重複**: 同じ内容を扱っている可能性があるチケット

#### 参照に関する注意事項

ファイルの移動や削除により、リンクが無効になる可能性があります。

- **移動時**: 参照元チケットのパスを更新する必要がある
- **削除時**: 参照が無効になるため、必要に応じて参照を削除するか、コメントとして残す

#### 例

```md
関連チケット
-------------------------

- 完了: ~~[setup-environment.md](../task/setup-environment.md)~~ - 環境構築(完了により削除)
- 関連: [implement-feature.md](../task/implement-feature.md) - 機能実装
```

リンク切れのリスクを最小化するため、重要な関連情報はチケット内に概要として記載することを推奨します。

参考資料の管理
-------------------------

チケットに関連する参考資料（ドラフト、スクリーンショット、既存ファイルのコピー等）は `_files/` ディレクトリで管理します。

### 参考資料ディレクトリの用途

以下のようなファイルを格納します。

- 作成途中のドラフト文書
- チケット作成時に参照した既存ファイルのコピー
- スクリーンショット、図表、画像ファイル
- サンプルコード、設定ファイルのスニペット
- その他、チケット理解に必要な補助資料

### ファイル命名規則

`{ticket-name}-{suffix}.{ext}` 形式を使用します。

#### 命名例

- `add-feature-draft.md`: add-feature.mdチケットのドラフト文書
- `fix-login-screenshot.png`: fix-login.mdチケットのスクリーンショット
- `improve-performance-sample.yml`: improve-performance.mdチケットのサンプル設定
- `design-api-diagram.drawio`: design-api.mdチケットの参考図表

#### 命名のポイント

- チケットファイル名（拡張子を除く）を先頭に配置
- サフィックスで用途を示す（draft, screenshot, sample, diagram等）
- 説明部分は英語・ハイフン区切りで簡潔に
- 拡張子は適切なものを使用

### チケットからの参照方法

チケット本文から相対パスでリンクします。

```md
参考資料
-------------------------

- [提案書ドラフト](../_files/add-feature-draft.md)
- [エラー画面スクリーンショット](../_files/fix-login-screenshot.png)
```

#### 複数の参考資料がある場合

1つのチケットに複数の参考資料を添付する場合、サフィックスで区別します。

```md
参考資料
-------------------------

- [要件定義ドラフト](../_files/add-feature-draft-requirements.md)
- [設計ドラフト](../_files/add-feature-draft-design.md)
- [画面モックアップ](../_files/add-feature-mockup.png)
- [サンプルコード](../_files/add-feature-sample.js)
```

#### 命名のポイント

- チケットファイル名の先頭部分を統一して使用
- サフィックスで用途・内容を区別
- 同じ種類のファイルが複数ある場合は、さらに詳細を追加（例: `screenshot-01.png`, `screenshot-02.png`）

### 参考資料の作成タイミング

以下のタイミングで参考資料を作成・配置します。

1. **チケット作成時**: 既存ファイルをコピーして参照する場合
2. **作業中**: ドラフトやスクリーンショットが必要になった場合
3. **チケット更新時**: 追加の参考資料が必要になった場合

### 参考資料のライフサイクル

#### 保持すべきファイル

- チケット理解に必須の資料
- 成果物の元となったドラフト
- 再現に必要なスクリーンショット

#### 削除を検討すべきファイル

- チケット完了後、正式な成果物に統合された一時ファイル
- 最終的に不要になったドラフト
- 他の場所に正式に配置されたファイル

#### 削除タイミング

**チケットクローズ時**: チケットをクローズする際、関連する参考資料の処理方法を確認します。

```
チケット「task/completed-task.md」をクローズします。
関連する参考資料の処理方法を選択してください。

1. 参考資料も削除する（推奨）
2. 参考資料もアーカイブする（チケットと一緒に保管）
3. 参考資料は削除せず、`_files/`に残す（他のチケットでも利用する場合）
```

- 参考資料の多くはチケット作業中の一時的な資料であり、チケット完了後は不要になります
- チケットクローズ前に、保持すべきファイルがないか確認します
- 長期的に保持すべき資料は、適切な場所 (ドキュメントディレクトリ等) に移動します
- この運用により、`_files/` ディレクトリの肥大化を防ぎ、管理性を維持します

### 参考資料管理の注意事項

- **ファイルサイズ**: 大きなファイルはGit管理対象から除外することを検討
- **機密情報**: パスワード、APIキー等を含めない
- **重複回避**: 既に別の場所で管理されているファイルのコピーは最小限に
- **整理整頓**: 定期的に不要なファイルを削除し、ディレクトリを整理

人間への確認が必要なケース
-------------------------

以下の場合は、必ずユーザーに確認を求めます。

### 情報不足

- 必須項目を埋めるための情報が不足している
- チケット種別の判断に必要な情報がない
- 背景や理由が不明確

### 判断の曖昧性

- 複数の種別に該当する可能性がある
- タスクの優先度や緊急度が不明
- 既存チケットとの重複の可能性がある

### 重要な決定

- 既存チケットの大幅な変更
- チケットの種別変更(ファイル移動)
- チケットの削除(完了・取りやめ)
- 複数のタスクへの分割

### 技術的・業務的判断

- セキュリティやパフォーマンスに影響する可能性がある内容
- 他のシステムやチームに影響を与える可能性がある内容
- 実現可能性について専門的な判断が必要な内容

ベストプラクティス
-------------------------

### チケット作成時

1. **一つのチケットには一つのトピック**: 複数の異なる内容を一つのチケットに詰め込まない
2. **具体的な記述**: 「あれ」「それ」などの代名詞を避け、具体的に記述
3. **背景の明確化**: なぜこのチケットが必要なのかを必ず記載
4. **完了条件の明示**: 特にタスクでは、何をもって完了とするかを明確に

### コミュニケーション

1. **確認のタイミング**: 情報不足を感じたら即座に確認
2. **選択肢の提示**: 判断が必要な場合は、2-3の選択肢を提示
3. **リスクの共有**: 潜在的な問題や制約があれば事前に伝える
4. **代替案の提案**: より良い方法がありそうな場合は提案

### 継続的な改善

1. **パターンの学習**: ユーザーの好みや傾向を学習し、提案の質を向上
2. **一貫性の維持**: 同様のケースでは同様の判断基準を適用
3. **フィードバックの活用**: ユーザーの修正や指摘を次回に活かす

ADRと共有リソースの活用
-------------------------

### `_shared/`ディレクトリ

`_shared/`ディレクトリには、チケット管理の文脈で参照される情報を格納します。
特に重要なのがADR(Architecture Decision Record)です。

### ADR(Architecture Decision Record)

ADRは、重要な技術的・アーキテクチャ的な意思決定を記録するためのドキュメントです。

#### ADRを作成すべきケース

- 技術スタックの選定(フレームワーク、ライブラリ、言語など)
- アーキテクチャパターンの採用(MVC、マイクロサービス、イベント駆動など)
- データベース設計の重要な判断
- セキュリティやパフォーマンスに関わる設計決定
- 大きな方向転換や方針変更

#### ADRの作成タイミング

- タスクやアイデアの議論の中で重要な技術判断が必要になった時
- 複数の選択肢があり、その選定理由を記録すべき時
- 将来、「なぜこの判断をしたのか」を振り返る必要がありそうな時

#### エージェントとしての対応

チケットの議論の中で以下のような状況に遭遇した場合、ADRの作成を提案します。

```
この判断は重要な技術選定に関わるものです。
ADR(Architecture Decision Record)として記録することをお勧めします。

### 理由

- 複数の選択肢がある
- 将来的に判断の根拠を参照する可能性が高い
- 他のチームメンバーにも影響を与える判断

_template/adr/example-YYYYMMDD.md を使用してADRを作成しますか？
```

### その他の共有リソース

ADR以外にも、以下のような情報を`_shared/`に格納できます。

- 環境情報やセットアップ手順
- よく参照される技術メモ
- チーム内の用語集
- 一時的なスクリプトやツール

各リソースタイプは、必要に応じてサブディレクトリに分類することで管理しやすくなります。

### `docs/`ディレクトリとの使い分け

```
.ticket/          # チケット管理システム
  └ _shared/      # チケット文脈で参照される情報、ADR
      └ adr/      # ADRを格納するサブディレクトリ
docs/             # プロジェクト全体の正式ドキュメント
```

#### `_shared/`に配置するもの

- ADR(意思決定の記録)
- チケット作業中に参照する補足情報
- 開発プロセスに関する情報

#### `docs/`に配置するもの

- 正式な設計書
- API仕様書
- ユーザーマニュアル
- システム構成図
- 運用手順書

#### 参照の仕方

```md
### チケット内から`_shared/`を参照
詳細は [ADR: データベース選定](../_shared/adr/database-selection-20230115.md) を参照してください。

### チケット内から`docs/`を参照
仕様については [API設計書](../../docs/api-specification.md) を参照してください。
```

### ADRファイルの命名規則

ADRは以下の命名規則に従います。

```
brief-description-YYYYMMDD.md
```

- `brief-description`: 意思決定内容を簡潔に表す英語表記
- `YYYYMMDD`: ADRを作成した日付(例: 20250115)
- 配置場所: `_shared/adr/`
- 例: `database-selection-20230115.md`, `build-system-webpack-20240315.md`

日付をサフィックスとして付与することで、ファイル名の衝突を回避しつつ、同じトピックのADRでも時系列を識別できます。
また、内容ベースの命名により、アルファベット順でソートした際に関連するADRが近くに配置されます。

#### 命名例: 同じトピックの変遷

```
_shared/adr/
  ├ build-system-gulp-20230201.md
  ├ build-system-vite-20250510.md
  └ build-system-webpack-20240315.md
```

このように、ビルドシステムに関する意思決定が時系列で並び、関連性が一目で分かります。

トラブルシューティング
-------------------------

### よくある問題と対処法

**Q: アイデアと要望の区別が難しい**
A: 「実現したい」という意思が明確かどうかで判断。思いつきレベルならアイデア、やりたいことが明確なら要望。

**Q: 大きなタスクをどう扱うべきか**
A: まず一つの親タスクとして記録し、ユーザーに「このタスクを複数の小タスクに分割しますか？」と確認。

**Q: 既存チケットとの重複が疑われる**
A: 重複の可能性をユーザーに伝え、既存チケットへのリンクと共に、新規作成・統合・既存更新のどれが良いか確認。

**Q: 技術的な実現可能性が不明**
A: その旨をチケットに明記し、調査タスクとして記録することを提案。

**Q: チケット削除後に参照したくなった場合(Git管理している場合)**
A: Git履歴から参照できることを説明。`git log --all --full-history -- path/to/file.md` などで確認可能。

**Q: ファイル移動時に他のチケットからの参照が壊れる**
A: 移動を実行する前に、他のチケットから参照されていないか確認し、必要に応じてパスを更新することを提案。

まとめ
-------------------------

このチケット管理システムは、柔軟性とシンプルさを重視しています。エージェントとして重要なのは。

- ユーザーの意図を正確に理解する
- 適切な種別とテンプレートを選択する
- 不明点は推測せず、確認する
- 一貫性のある記録を維持する
- チケット間の関係性を明確にする
- ライフサイクルに応じてファイルを適切に移動・削除する

これらの原則に従うことで、ユーザーにとって有用なチケット管理システムの運用を支援できます。
